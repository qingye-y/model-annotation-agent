> **版本**：v2.0 | **日期**：2026-05-24 | **编写**：小B

---

## 版本记录

| 版本 | 日期 | 变更摘要 |
|------|------|---------|
| v2.0 | 2026-05-24 | 去右侧边栏（弹窗分配）；分配记录可跳转到标注任务页；分配记录可删除；验收标准 |

---

## 1. 问题

当前 dispatch_center.html 设计过于复杂：
- 右侧边栏占用空间但几乎不显示内容（截图显示"请从左侧选择一个规则进行分配"占了大半屏）
- 分配操作已有弹窗，右侧边栏冗余
- 最近分配记录只读不可操作，无法跳转追溯、无法删除

---

## 2. 改动

### 2.1 去掉右侧边栏

| 改前 | 改后 |
|------|------|
| 左右分栏：左侧任务池表格 + 右侧分配面板（占 40% 宽度） | 全宽单栏：任务池表格满屏 |

分配操作全部通过点击表格行的「分配」按钮 → 弹窗完成。

### 2.2 分配记录可跳转

「最近分配记录」列表中每条记录新增「查看任务」按钮：

| 点击 | 行为 |
|------|------|
| 「查看任务」 | 跳转到 annotation_list.html?rule=xxx&date=yyy&instance=zzz |
| 标注任务页 | 读取 URL 参数，筛选只显示该规则+日期+实例对应的任务（可能是多个） |

### 2.3 分配记录可删除

每条分配记录右侧新增「删除」按钮：

| 操作 | 效果 |
|------|------|
| 点击删除 | 二次确认 → 调 `DELETE /api/dispatch/assign-log/{id}` → 刷新列表 |
| 后端 | 删除 dispatch_log 表对应记录（不影响已分配的 Annotation 和 RawData） |

---

## 3. API 变更

| Method | Path | 说明 | 状态 |
|--------|------|------|------|
| DELETE | `/api/dispatch/assign-log/<id>` | 删除分配记录 | 新增 |
| GET | `/api/dispatch/assign-log` | 已有，列表接口不变 | — |

annotation_list.html 读取 URL 参数：

| 参数 | 作用 |
|------|------|
| `rule` | 筛选该规则名 |
| `date` | 筛选该数据日期 |
| `instance` | 筛选该实例 |
| 三个参数都传 | 只显示匹配的任务 |

---

## 4. 前端改动（dispatch_center.html）

| 区域 | 改动 |
|------|------|
| 右侧分配面板 | 删除 `.dispatch-edit-panel` 及其所有 JS 绑定 |
| 左侧容器 | `max-width` 约束去掉，宽度 100% |
| 分配按钮 | 点击 → 弹窗（已有 `executeAssign` 逻辑，不需要右侧面板预处理） |
| 分配记录列表 | 每条追加两个按钮：`<a>` 跳转标注页 + `🗑` 删除 |
| 删除确认 | `confirm("确定删除？")` → `fetch DELETE` |

---

## 5. 验收标准

| # | 验证项 | 预期 |
|---|--------|------|
| AC-1 | 页面全宽 | 右侧无分配面板，表格满屏 |
| AC-2 | 点击分配 → 弹窗 | 弹窗正常显示标注员选择/数量/方式，提交成功 |
| AC-3 | 分配记录有「查看任务」 | 点击 → 跳转到标注任务页，URL 含 rule/date/instance 参数 |
| AC-4 | 标注任务页按参数筛选 | 只显示匹配的任务，可以是一个或多个 |
| AC-5 | 分配记录有「删除」 | 点击删除 → 确认 → 记录消失 → 刷新后不重现 |
| AC-6 | 删除不影响数据 | 已分配的 Annotation/RawData 不受影响 |
| AC-7 | 无任务时不报错 | 0 分配记录时显示"暂无分配记录" |
